# 工作流：跨仓库数据依赖

适用于网站功能依赖另一个工程生成的数据、静态文件或后台服务的任务。本项目主要对应 `snh48-fan-hub` 生成的时光轴、直播回放和图片代理数据。

## 1. 先确认数据边界

- 找到数据生产方、消费方和线上路径。
- 阅读消费方的 `project_profile.md`，再读生产方的数据对接文档。
- 区分代码部署和数据同步：代码通过 GitHub，运行数据按项目 profile 指定的同步链路走。
- 确认哪些数据是网站运行必需，哪些是生产方内部数据、缓存、日志或大体积原始素材。
- 如果任务涉及腾讯云到阿里云同步，确认当前方向是阿里云主动拉取；不要沿用旧的腾讯云主动推送检查命令。

## 2. 变更前说明

涉及多项变更时，先一次性列出：

- 预期效果。
- 可能坏处。
- 是否建议实施。
- 需要同步的仓库、服务器和数据路径。

默认等待用户确认；用户明确表示全部实施时再批量推进。

## 3. 实施顺序

优先顺序：

1. 更新数据生产方的数据结构、脚本或文档。
2. 更新消费方读取逻辑、环境变量示例或部署文档。
3. 更新两个仓库的对接说明，特别是字段、路径、同步方向和验收命令。
4. 如果变更同步策略、频率或目录，同时更新消费方 `AGENTS.md`、`doc/daily_website_check.md`、`doc/running_status.md`、`doc/security/security_baseline.md` 和 project profile。
5. 分别提交并推送到对应 GitHub 仓库。
6. 先同步生产方服务器，再同步消费方服务器。
7. 如果有运行数据变更，再执行 profile 指定的数据同步。

不要用一个仓库的部署动作覆盖另一个仓库的运行数据。

## 4. 验证

按任务目标选择验证项：

- 生产方：脚本语法检查、关键命令 dry-run、后台日志、目标数据文件存在且格式正确。
- 消费方：目标页面/API 能读取新数据，降级路径正常，日志无明显错误。
- 同步链路：目标服务器上的 Git 提交、目标数据路径和必要文件时间戳符合预期；涉及腾讯云到阿里云时，还要验证阿里云拉取 cron 和日志正常、腾讯云旧推送 cron/进程停用、稳定小文件 hash 一致、动态小数据只有合理分钟级延迟；如果使用分组同步，动态变化时应优先只触发 `groups=dynamic`。

## 5. 失败处理

- 生产方数据格式不兼容：停止消费方部署，先修复数据契约或兼容层。
- 远端 Git 有本地运行数据改动：不要强行覆盖，先确认这些数据是否需要提交、备份或保留。
- 数据同步失败：保留源服务器数据，报告失败命令和目标路径，不删除目标已有数据。
