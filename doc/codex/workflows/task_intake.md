# 工作流：任务进入与确认方式

目标：先把任务类型、风险、确认方式和完成标准说清楚，避免把一次任务的验收项误当成所有任务的通用标准。

## 1. 任务分类

开始前先判断任务属于哪类：

| 类型 | 例子 | 主要关注 |
|------|------|----------|
| 安全审计 | 查攻击面、加安全头、收敛公网端口 | 风险、兼容性、文档、线上验证 |
| 安全修复 | CSP、XSS、限速、Cookie、代理 IP | 修复收益、坏处、回归风险 |
| 功能 bug | 页面报错、API 异常、数据不显示 | 复现、最小修复、回归测试 |
| 新增功能 | 新页面、新 API、新管理能力 | 需求边界、数据模型、用户流程 |
| 部署同步 | GitHub 到服务器、重启服务 | 目标环境、提交、服务状态、烟测 |
| 文档维护 | 更新 README、安全基线、运维手册 | 与代码一致、可操作、不过度泛化 |

## 2. 多项措施的说明格式

如果一次涉及多项措施，先列出：

| 措施 | 预期效果 | 可能坏处 | 建议 |
|------|----------|----------|------|
| 示例 | 降低某风险 | 可能影响某功能 | 建议/可选/暂缓 |

默认等用户逐项确认。若用户明确说“按推荐全部推进”，可以批量执行。

## 3. 完成标准定义

每次任务要单独定义 Done when：

- 代码任务：相关检查通过，改动范围清楚。
- 功能任务：目标功能路径和关键回归点通过。
- 安全任务：本次安全目标通过，安全文档更新。
- 部署任务：指定环境同步到目标提交，服务重启或重载，任务相关烟测通过。

不要把某个任务的专项检查扩展成所有部署的固定完成标准。部署验收项应来自本次任务目标、变更范围和用户明确要求。

## 4. 何时需要用户操作

这些事项通常需要用户确认或自行操作：

- 云控制台安全组、证书、DNS、备案信息。
- 真实 `.env` 密码和 API key。
- 可能影响可用性的 Nginx、数据库、服务重启。
- 需要付费、外部账号授权、删除数据、回滚历史的操作。

能在代码里先做的，先改 `.env.example`、文档或脚本；再告诉用户需要在服务器或云控制台同步什么。
