# 工作流：通过 GitHub 同步部署

适用于把本地已完成的修改同步到一台或多台服务器。具体服务器和命令见 `doc/codex/project_profile.md`。

## 1. 部署前检查

- `git status --short`，确认没有意外文件。
- 运行本项目本地验证命令。
- 如果改了 JS/CSS，确认已重建并暂存 dist。
- 如果新增或修改 `.env` 项，确认 `.env.example` 已更新，并在部署计划或执行前明确提醒用户按 `.env.example` 检查目标服务器 `.env`。
- 如果改了 Nginx，准备远端 `nginx -t` 和 reload。

## 2. 提交与推送

```bash
git add .
git commit -m "<message>"
git push origin <branch>
```

提交信息应描述任务结果，不只写 “update”。

## 3. 远端同步

多环境分阶段发布：

- 如果项目 profile 指定了先验环境或用户要求先验证某一台服务器，先只同步该环境。
- 先验环境完成机器烟测后，向用户报告提交、服务状态、目标 URL 和已验证项，并明确说明尚未同步其他环境。
- 只有用户明确确认先验环境可接受后，才继续同步后续环境或执行数据同步。
- 未获得确认前，不要用 `deploy all` 绕过分阶段验收；`deploy all` 只适合用户明确要求一次性同步，或本次变更确认为无用户可见影响。

运行数据同步策略变更：

- 先读 `doc/codex/project_profile.md` 的数据生成工程依赖段落，确认同步方向、源路径、目标路径、cron 所在服务器和手动兜底入口。
- 如果修改腾讯云到阿里云的数据同步方式，必须同步更新每日检查、运行状态、安全基线和 AGENTS 规则，不要只改脚本。
- 部署或切换后同时验证新任务已启用、旧任务已停用、稳定小文件两端 hash 一致、动态小数据只有合理分钟级延迟；如果使用分组同步，动态变化时应优先只触发 `groups=dynamic`。
- 不要把腾讯云 `sync-to-aliyun-loop.sh` 或 `sync-to-aliyun-if-changed.sh` 恢复为生产自动任务；腾讯云推送只作为用户确认后的临时手动兜底。

在目标服务器：

- 查看当前 `git status --short`，识别运行时文件和真实冲突。
- 执行 `git pull`。
- 如果本次新增或修改环境变量，先检查目标服务器 `.env` 是否已按 `.env.example` 补齐；本项目可用 `python3 deploy/deploy.py check-env all` 只比较键名。缺少新增的非敏感生产项时，先增量补齐或提醒用户补齐，再让配置生效。不要输出真实密码。
- 按本次变更范围判断是否需要重启应用，不要把重启作为所有部署的固定动作。
- 如果 Nginx 配置变更，复制对应配置、运行 `nginx -t`、reload。

重启判断：

| 变更范围 | 是否重启 Python 服务 | 说明 |
|---|---|---|
| 仅文档、Codex 规则、部署说明 | 不需要 | 远端 `git pull` 后确认目标提交即可 |
| 仅静态资源产物、图片、前端 JS/CSS | 通常不需要 | Nginx 直接读取文件；源 JS/CSS 修改仍需先构建并提交 dist |
| 仅模板 HTML | 通常不需要 | 刷新目标页面验证；如果同时改了 Python 上下文或路由则需要重启 |
| 确认可按请求读取的数据文件 | 通常不需要 | 例如已知的时光轴运行数据；不确定是否启动时加载时按需要重启 |
| Python 代码、依赖、入口、后台任务 | 需要 | 服务进程已加载旧代码 |
| `.env` 或服务运行参数 | 需要 | 先按 `.env.example` 增量补齐，再重启 |
| Nginx 配置、证书、CSP | 不重启 Python；需要 Nginx reload | 必须先 `nginx -t`，失败则不要 reload |

本项目多服务器工具支持无重启同步：

```bash
python3 deploy/deploy.py deploy all --no-restart
```

Nginx 配置变更可用：

```bash
python3 deploy/deploy.py deploy all --nginx --no-restart
```

## 4. 验证

通用部署验证：

- 远端 Git 提交为目标提交。
- 服务为 active 或进程存在。
- 本机后端接口或首页可访问。
- 线上目标页面/API 烟测通过。
- 如果本次重启了服务，启动日志无明显错误。

任务特定验证：

- 功能任务：验证目标功能和关键回归点。
- 安全任务：验证本次安全目标，例如安全头、公网端口、CSP、Cookie、限速。
- Nginx/env/网络任务：验证实际配置已生效。

不要把安全任务的具体验收项固定成所有部署任务的通用完成标准。

## 5. 运行时文件处理

服务器可能存在日志、缓存、用户数据、限速状态等未跟踪文件。除非用户明确要求，部署时不要删除它们。

## 6. 失败处理

- `git pull` 冲突：停止，报告冲突文件，不强行覆盖。
- 服务启动失败：查看日志，优先修复配置或代码错误。
- `nginx -t` 失败：不要 reload，保留当前线上配置，报告错误。
- 线上烟测失败：保留日志和状态，说明影响范围；必要时按用户确认回滚。
