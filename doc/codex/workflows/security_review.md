# 工作流：网站安全审计与修复

适用于安全分析、安全修复、上线前安全复核。跨项目可复用，具体命令从 `doc/codex/project_profile.md` 读取。

## 1. 先建立范围

确认本次目标：

- 是只做代码审计，还是要部署到服务器？
- 涉及哪些入口：页面、API、Nginx、静态资源、后台管理、第三方服务？
- 是否允许联网验证线上环境？
- 是否需要同步多台服务器？

## 2. 攻击面清单

至少检查：

- 鉴权：公开端点、管理端点、异步结果读取、Cookie。
- 输入输出：用户输入、CSV/第三方数据、LLM 输出、`innerHTML`、URL 和 class 拼接。
- 速率限制：会消耗费用、写日志、提交表单、登录尝试的端点。
- 网络边界：公网端口、反向代理、真实 IP 获取、`X-Forwarded-For` 信任边界。
- 浏览器安全：CSP、HSTS、X-Frame-Options、X-Content-Type-Options、Referrer-Policy。
- 供应链和静态资源：CDN、外部脚本、source map、构建产物。
- 日志和隐私：密码/API key、邮箱、IP、错误堆栈、第三方 API 数据。
- 部署一致性：多云配置、Nginx 配置是否同步、`.env.example` 是否覆盖新增项。

## 3. 修复前说明

多项修复时先给用户一张表：

| 修复项 | 效果 | 坏处/兼容性风险 | 建议 |
|--------|------|----------------|------|

只有得到确认后再实施；用户确认“全部推进”时可以批量执行。

## 4. 实施规则

- 优先按现有架构和 helper 修改，避免大重构。
- `.env` 新变量先写入 `.env.example` 和文档；在让配置生效前，明确提醒用户按 `.env.example` 检查服务器 `.env`。
- Nginx 变更同步维护所有环境配置。
- 前端 JS/CSS 源码改动后重建 dist。
- 安全措施的取舍必须写入 `doc/security/security_baseline.md` 或对应安全文档。

## 5. 验证

本地验证：

- Python 编译或测试。
- JS/CSS 构建和语法检查。
- `git diff --check`。
- 受影响 API/页面的最小验证。

线上验证按本次目标选择：

- Nginx 语法和实际加载配置。
- 目标页面/API 是否正常。
- 安全头是否存在。
- 公开端口是否符合本次安全目标。
- 第三方资源、HLS、图片代理等兼容性。

## 6. 输出

最终回复包含：

- 已修复什么。
- 如何验证。
- 还剩哪些只能靠用户操作或长期注意的事项。
- 是否已部署到哪些环境。
